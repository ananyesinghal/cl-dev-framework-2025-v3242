1. Overview

This framework provides a systematic methodology for evaluating HTML deliverables from the development team. It is designed to:

  • Catch technical issues before client delivery
  • Identify patterns that reveal skill gaps
  • Inform training and hiring decisions
  • Maintain consistent quality standards across all projects

Technical Correctness Valid HTML, CSS, JavaScript ‚Äî code that works without errors
Best Practices Semantic markup, maintainable code, performance, security

2. QA Principles

2.1 Evidence-Based Only

Every issue must cite:

  • Exact code (minimum 10 characters of context)
  • Line number
  • Specific rule being violated

If you cannot quote the problematic code, do not flag it as an issue.

2.2 Context Matters

Before flagging an issue, consider if there's a valid reason for the pattern:

  • Framework conventions (React, Vue, Angular)
  • Intentional design choices
  • Performance optimizations with comments
  • Browser-specific fixes
  • Progressive enhancement patterns

2.3 Pattern Recognition

Individual issues matter less than patterns:

  • One missed alt tag = a mistake
  • Five missed alt tags = a training need
  • Consistent accessibility misses = mindset problem

2.4 Systematic Coverage

Check every rule, every time. Skipping sections leads to missed issues. The framework exists to prevent oversight, not to be used selectively.

2.5 Avoiding False Positives

Do NOT flag as issues:

  • Valid HTML5 patterns
  • Framework-specific conventions
  • Intentional accessibility patterns (aria-hidden on decorative elements)
  • Performance optimizations with explanatory comments
  • Browser compatibility fixes with comments/prefixes
  • Empty alt="" on confirmed decorative images
  • Code clearly marked as development/prototype stage

11. Developer Evaluation

11.1 Purpose

QA findings are not just about fixing code ‚Äî they reveal how a developer thinks. Patterns in errors indicate gaps in knowledge, attention, or mindset that can be addressed through targeted training or feedback.

11.2 Pattern Analysis

Pattern What Repeated Misses Indicate Action
Accessibility Developer doesn't consider users with disabilities as "real" users Empathy training, exposure to assistive technology
Semantic HTML Developer learned HTML as "what looks right" not "what is right" Fundamentals training on document structure
Security Developer doesn't think adversarially Threat modeling training, secure coding practices
Content Quality Developer treats content as "someone else's problem" Clarify that content is part of deliverable
Performance Developer doesn't consider page load as UX Core Web Vitals training, user perception
CSS Quality Developer uses CSS to "make it look right" Training on cascade, specificity, maintainability

11.3 Scoring Rubric

BASE SCORE CALCULATION:
= (Number of Critical Rules √ó 2) + (Number of Important Rules √ó 1) + (Number of Recommendation Rules √ó 0.5)

DEDUCTIONS:
- Critical rule failed:      -2 points each
- Important rule failed:     -1 points each
- Recommendation rule failed: -0.5 points each

MINIMUM SCORE: 0

Score Interpretation

Note: Scores are calculated as a percentage of the maximum possible score based on the total number of rules.

Score % Rating Meaning
90-100% Excellent Ready for client delivery
75-89% Good Minor revisions needed
50-75% Needs Work Significant issues to address
0-50% Unacceptable Requires full rework and discussion

11.5 Feedback Guidelines

When providing feedback to developers:

  1. Be specific ‚Äî Quote exact code, explain why it's an issue, show the fix
  2. Focus on patterns ‚Äî One miss is a mistake; five misses is a training need
  3. Explain the "why" ‚Äî Developers learn better when they understand impact
  4. Acknowledge growth ‚Äî When a developer fixes a previous pattern, recognize it
  5. Separate skill from effort ‚Äî Not knowing is fixable; not caring is a different problem
